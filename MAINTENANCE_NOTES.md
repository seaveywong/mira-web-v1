# Mira 中优先级备忘

更新时间：2026-05-27

## 中优先级

- 前端 `frontend/index.html` 体积过大，后续应按页面拆成账户、素材、铺广告、设置、守护模式等模块。
- `api/accounts.py` 同时承担导入、Token、主页库、像素、状态同步，后续应拆分路由，降低改一处影响全局的风险。
- 为余额显示、账户状态码、铺广告预检、预热扫描、镜像快照增加自动化测试。
- 预热和铺广告可抽出共用的创建 Campaign/AdSet/Ad 参数构建器，减少 Meta API 参数漂移导致的重复修复。
- 守护健康面板后续可接 Telegram 异常摘要，避免只在前端设置页查看。
- 主页库状态增加过期复查机制，例如超过 24 小时自动重新探测 `is_published`、`tasks`、`page_can_advertise`，避免用到已失效主页。
- 预热失败原因按类型归类展示：主页不可投放、账户不可写、Token 权限、素材问题、定位问题、FB 临时错误。
- 预热扫描结果增加账户级跳过原因列表，让每个账户为什么开始/跳过/失败都能直接看懂。
- 主页库前端默认仅显示可投放主页，增加“显示全部/仅可投放”过滤，减少误选受限或未发布主页。
- 素材使用记录继续补细：记录预热/铺广告使用过哪些素材、在哪些账户成功或失败、失败原因是什么。

## 下次优先修正

- 多素材单账户铺广告：已确认暂不做。当前保持“单素材可铺单账户/多账户”的使用模型，避免把素材维度和账户维度混在一个入口里增加误操作。
- 受众补齐策略：下一轮重点做“尽量补齐每个 AdSet 的有效受众”。当某组兴趣词被 FB 判定合并、废弃或不可用时，优先使用 FB 返回的替代兴趣、素材 AI 兴趣池里的备用兴趣、同素材历史可用兴趣进行补位；只有补位失败时才降级宽泛，并明确标记。
- 受众覆盖目标：每次铺广告前先计算目标 AdSet 数、每组期望兴趣数、可解析兴趣数和备用兴趣数；如果无法满足目标覆盖，前端预检提示“受众不足/将降级 X 组”，由用户决定是否继续。
- 受众更新策略：每次铺广告会按素材 AI 兴趣词、账户、日期生成 rotation seed，理论上会轮转兴趣顺序；下一轮要把“解析到 X 个、使用 X 个、补齐 X 个、降级 X 组”写入结果，确保能判断受众是否真的更新。
- 铺广告受众降级可见性：当前 `120251815072000399` 这类 AdSet 若兴趣词被 FB 判定无效，会自动改成宽泛受众继续创建；但本地 `targeting_json` 仍可能记录原兴趣词。下一轮要让 `_create_adset` 返回最终 targeting/adset name，并在 DB/前端明确标记“已补齐/已降级宽泛”，避免后台看到空受众但 Mira 仍显示兴趣受众。
- 铺广告结果对账与展示：为每次创建完成后增加 FB AdSet targeting 回读，对比本地预期 targeting。前端按 AdSet 展示“兴趣受众/补齐受众/宽泛受众/降级宽泛”标签，并显示预期兴趣、实际生效兴趣、被替换/移除原因。
